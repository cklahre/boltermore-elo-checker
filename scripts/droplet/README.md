# Discord bot on a DigitalOcean droplet

Layout on the server:

| Path | Purpose |
|------|---------|
| `/opt/eloevent/repo/` | Mirror of this repo from GitHub Actions (`rsync`), excluding `.git`, workflows, `bcp-matches.json`, and CI-built binaries |
| `/opt/eloevent/bin/eloevent-bot` | Discord bot binary (deployed by CI) |
| `/opt/eloevent/bin/local-elo` | Rebuild `leaderboard.json` from matches (deployed by CI) |
| `/opt/eloevent/data/bcp-matches.json` | Large match export — **not** in git; you provide it on the droplet |
| `/opt/eloevent/data/leaderboard.json` | Generated by `refresh-leaderboard.sh` |
| `/opt/eloevent/env/bot.env` | `DISCORD_BOT_TOKEN`, `DISCORD_GUILD_ID` — **rewrite on every deploy** from GitHub secrets (or create locally before first CI deploy) |
| `/opt/eloevent/env/refresh.env` | Optional: `UPDATE_MATCHES_CMD` to fetch new matches before Elo run |
| `/opt/eloevent/scripts/refresh-leaderboard.sh` | Weekly / manual refresh |

## One-time droplet setup

1. Install Go is **not** required on the droplet if you only use CI-built binaries.
2. Clone this repo (or copy `scripts/droplet/` onto the box) and run:

   ```bash
   sudo bash scripts/droplet/install-on-droplet.sh
   ```

3. If you are **not** using CI to write Discord env yet, edit `/opt/eloevent/env/bot.env` and `chmod 600` it (deploy overwrites this file when `DISCORD_*` secrets are set).
4. Place your match export at `/opt/eloevent/data/bcp-matches.json` (see “Large matches file” below).
5. Run `/opt/eloevent/scripts/refresh-leaderboard.sh` once, then `systemctl start eloevent-discord-bot` (or rely on CI after pushing to `main`).

## GitHub Actions

Required repository **secrets** (names wired in the workflows):

| Secret | Role |
|--------|------|
| `SSH_KEY` | SSH private key for the droplet (full PEM/OpenSSH block, including newlines). |
| `DO_HOST` | Droplet hostname or IP (`ssh-keyscan`, `scp`/`ssh` target). |
| `DISCORD_BOT_TOKEN` | Discord bot token; written into `/opt/eloevent/env/bot.env` on each deploy. |
| `DISCORD_GUILD_ID` | Guild snowflake for slash registration; same file as above. |

Optional repository **variable** `DO_SSH_USER`: SSH username (defaults to **`root`** if unset).

Nothing else is required for the two workflows—no personal access token, PAT, or `GITHUB_TOKEN` access beyond checkout.

- **Deploy on push to `main`**: `.github/workflows/deploy-discord-droplet.yml` builds Linux `amd64` binaries, **`rsync`s the checkout to `/opt/eloevent/repo/`** (same excludes as below), rsyncs binaries plus scripts, copies the systemd unit, restarts `eloevent-discord-bot`. Pushes that only touch `site/**` are ignored.

Repo sync excludes: `.git/`, `.github/`, `.cursor/`, **`bcp-matches.json`** (keep only on-disk under `/opt/eloevent/data/`), and the ephemeral **`eloevent-bot`** / **`local-elo`** build artifacts (those land under **`bin/`** instead). The sync uses **`--delete`**, so anything you add only under **`/opt/eloevent/repo/`** and not in git will be removed on the next deploy.

- **Weekly leaderboard refresh**: `.github/workflows/refresh-leaderboard-droplet.yml` runs Mondays **12:00 UTC** (~7 AM US Eastern **standard** time; during daylight saving, that is ~8 AM local). GitHub cron is UTC-only; for exactly “America/New_York wall clock,” use a systemd timer **on** the droplet instead of Actions.

Both workflows SSH in and expect `/opt/eloevent` dirs to exist (`install-on-droplet.sh` creates them).

## Large `bcp-matches.json`

Do **not** commit it. Typical options:

1. **Copy once**, then optionally refresh manually:  
   `rsync -avP ./bcp-matches.json root@YOUR_DROPLET:/opt/eloevent/data/`
2. **Automate pull on the droplet**: set `UPDATE_MATCHES_CMD` in `/opt/eloevent/env/refresh.env` to `curl`/`wget` from a private URL (e.g. DigitalOcean Spaces with a scoped token), or run your existing export script after checking out tooling on the droplet.
3. **GitHub Actions** should only run `refresh-leaderboard.sh` over SSH; do not upload gigabytes through Actions unless you add a deliberate artifact step.

Weekly refresh runs `local-elo` **on** the droplet against the matches file already there (or fetched by `UPDATE_MATCHES_CMD`).
